# Claude Code 系列教程知识文档

> 项目：Net Ninja《Claude Code Tutorial》系统学习
> 范围：正式教程 10 集 + 1 条宣传短片的路线价值判断
> 产出目标：把“逐集内容”升级成“可复用知识体系”

---

## 1. 先说结论

这套教程真正讲的，不是“怎么让 Claude Code 更会写代码”，而是：

**怎么把 Claude Code 变成一个可控、可审、可复用、可接入工程流程的开发协作者。**

它的完整进阶路径可以压缩成一句话：

> **先接入工作流，再建立项目记忆，再管好上下文；在此基础上控制权限、引入规划、固化命令、扩展 MCP、使用 subagents，并最终把它接入 GitHub 协作流。**

换句话说，Claude Code 的成熟用法不是“更会 prompt”，而是：

1. **有边界**：权限、作用域、任务范围清楚
2. **有记忆**：项目规则不是每次口头重复
3. **有上下文治理**：不会把旧任务噪声一路拖下去
4. **有流程**：常见操作被固化为命令和 SOP
5. **有分工**：实现、审查、测试可以角色隔离
6. **有人负责**：最终判断仍然在人，不在 AI

## 2. 全 10 集知识地图

### 第一阶段：打底座（1-3）

#### #1 Introduction & Setup
重点：
- Claude Code 是终端里的 agentic coding tool
- 它更像“工作流入口”，不只是编辑器里的补全工具
- 正确姿势是人主导、AI 协作、小步执行、持续 review

#### #2 CLAUDE.md Files & /init
重点：
- `/init` 是新项目接入 Claude Code 的标准起手动作
- `CLAUDE.md` 是项目级运行说明书 / memory file
- 共享规则、项目本地偏好、全局个人偏好要分层管理

#### #3 Context
重点：
- Claude Code 的表现高度依赖上下文质量
- 上下文不是越多越好，而是越贴当前任务越好
- `/clear`、`/compact`、新开 session 等能力，本质上是上下文治理工具

### 第二阶段：进入可控执行（4-5）

#### #4 Tools & Permissions
重点：
- Claude 会自动选择工具，但你必须理解授权边界
- read / edit / bash 的风险不同
- 会话级加速和项目级白名单是两种不同的控制层

#### #5 Planning & Thinking
重点：
- Planning：解决“改动范围大、多文件协作”的问题
- Thinking：解决“逻辑复杂、需要先想清楚”的问题
- 计划不能替代边界说明；AI 会合理扩张任务范围

### 第三阶段：把经验固化成流程（6-7）

#### #6 Slash Commands
重点：
- Slash command 是把高质量 prompt + 规则 + 上下文固化成可复用入口
- 自定义命令不是花活，而是项目工作流资产
- 命令要同时写“要做什么”和“不要做什么”

#### #7 MCP Servers
重点：
- MCP 让 Claude Code 接入外部世界：最新文档、浏览器、数据库、API
- Context7 代表“最新文档能力”，Playwright 代表“浏览器执行/验收能力”
- MCP 不是装了就完，要配 scope、看状态、点名使用

### 第四阶段：进入多角色与团队协作（8-9）

#### #8 Subagents
重点：
- Subagent 的价值是职责隔离、上下文隔离、工具隔离
- 最适合承担“窄而深”的专项任务
- 最佳模式是：实现者与审查者分离

#### #9 Claude Code with GitHub
重点：
- Claude Code 可以进入 GitHub 协作流：issue -> branch -> PR -> review
- 自动处理 issue、自动 review PR，适合边界清晰的小任务
- 自动化 review 不是最终裁决，人仍要把关

### 第五阶段：治理与收束（10）

#### #10 Final Thoughts & Tips
重点：
- 永远自己检查代码
- 新 feature 建议新 session，保持上下文清洁
- AI 是协作者，不是思维替身
- 终端快捷操作本质上也是生产效率的一部分

## 3. Claude Code 的 10 个核心认知

### 1）它首先是“工作流工具”，其次才是“聊天工具”
Claude Code 最有价值的地方，不在于它能回你一句聪明话，而在于它能进入：
- terminal
- git
- 测试命令
- 项目结构
- review 流程
- GitHub workflow

所以它应该被当成“工程协作者”，而不是“对话机器人”。

### 2）真正放大效果的是结构化信息，不是神 prompt
Claude Code 的输出质量，更多取决于：
- 项目记忆是否清楚
- 当前上下文是否干净
- 任务范围是否收敛
- 边界条件是否明确

而不是你这次 prompt 语气多么华丽。

### 3）`CLAUDE.md` 是协作底座
`CLAUDE.md` 最重要的价值是减少反复解释。

建议至少写清楚：
- 项目结构
- 常用运行/测试命令
- 命名规范
- 组件/Hook/数据/样式/测试约束
- 哪些事情默认不要做
- 哪些情况要先问再动

### 4）上下文治理决定长期表现
如果你在一个会话里连续混聊：
- 购物车逻辑
- 样式微调
- 路由改造
- MCP 接入

Claude 很容易把旧目标、旧文件、旧假设一起带到新任务里。

所以推荐默认实践：
- **一个 feature 一个 session**
- 同一任务过长时先 `/compact`
- 任务切换时果断 `/clear` 或重开

### 5）权限控制是能力，不是负担
Claude 会自动选工具，但你要自己知道：
- 什么可以默认放开
- 什么要保留审批
- 什么只适合小范围白名单

推荐原则：
- `read`：低风险
- `edit`：中风险，建议默认保留审查
- `bash` / 外部副作用工具：高风险，按命令或 scope 限制

### 6）Planning 和 Thinking 不能混用
- **Planning**：面向“任务跨度大、改动点多”
- **Thinking**：面向“逻辑复杂、需要先推演”

最稳妥的实践：
- 大范围改动先 planning
- 难逻辑问题先 thinking
- 任何时候都别跳过 review

### 7）Slash Command 的本质是“把成熟流程产品化”
自定义命令不是为了少打字，而是为了把你的高频开发动作沉淀成：
- 统一入口
- 固定约束
- 固定上下文
- 固定验收要求

它应该被视为项目资产，而不是私人技巧。

### 8）MCP 让 Claude 从“会改代码”进化到“会接环境”
MCP 的意义不只是“多几个工具”，而是让 Claude 可以：
- 查最新框架文档
- 打开浏览器验收页面
- 读取数据库结构
- 操作服务 API

这会让它从“基于旧知识猜”转向“基于实时环境做”。

### 9）Subagent 的价值不是并行，而是分工
最成熟的 subagent 用法不是“多开几个 AI 一起卷”，而是：
- 让不同角色各做自己的事
- 每个角色拿到自己的上下文
- 每个角色只拿必要工具

典型分工：
- 实现 agent
- 测试 agent
- 安全审查 agent
- UX reviewer agent

### 10）GitHub 集成意味着 Claude 进入正式协作流
这是很关键的分水岭。

一旦接到 GitHub：
- Claude 不再只是本地副驾驶
- 它开始成为仓库工作流的一部分
- 但责任边界仍然要保留在人工 review 上

## 4. 推荐的 Claude Code 使用框架

## 第一层：接入层
目标：让 Claude Code 安全进入你的开发环境

推荐动作：
1. 在项目目录内启动 Claude Code
2. 先让它总结项目，再要求改代码
3. 先切独立分支，再进行 AI 试验
4. 保持 diff 可见、可回滚

## 第二层：记忆层
目标：让 Claude 懂项目，而不是每次猜

推荐动作：
1. 执行 `/init`
2. 审核并补充 `CLAUDE.md`
3. 把高频规范写进 memory，而不是每次口头重复
4. 区分 project / personal / global 规则

## 第三层：上下文层
目标：只给当前任务真正需要的信息

推荐动作：
1. 尽量用 `@文件` 精确指定输入
2. 能给片段就不一定给整文件
3. 任务切换时清理历史
4. 一个 feature 一个会话

## 第四层：执行层
目标：让 Claude 在可控边界内真正落地

推荐动作：
1. 按权限等级开放能力
2. 大范围任务先 planning
3. 难逻辑任务先 thinking
4. 每一步都保留人工 review

## 第五层：流程层
目标：把经验变成可重复执行的流程

推荐动作：
1. 用 slash commands 固化高频任务
2. 在命令里写清正向要求与边界限制
3. 把命令视为项目级工作流资产

## 第六层：扩展层
目标：接入外部真实环境

推荐动作：
1. 用 Context7 拉最新文档
2. 用 Playwright 做浏览器验收
3. 用项目级 MCP scope 共享关键能力
4. 把“何时必须查文档”写进项目规则

## 第七层：协作层
目标：构建多角色协作系统

推荐动作：
1. 用 subagents 做职责分离
2. reviewer 不默认拥有编辑权
3. 把 subagent 嵌入固定命令/流程
4. 明确“谁实现、谁审查、谁验收”

## 第八层：团队层
目标：接入 GitHub 协作流

推荐动作：
1. 让 AI 处理边界清晰的小 issue
2. 自动 review PR，但不自动外包判断
3. 限制授权仓库范围
4. 阅读并理解生成的 workflow 文件

## 5. 实战 SOP（建议版）

### SOP A：新项目接入 Claude Code
1. 进入项目目录
2. 启动 Claude Code
3. 让它总结项目结构
4. 执行 `/init`
5. 审核并补充 `CLAUDE.md`
6. 切独立分支
7. 从小任务开始试跑

### SOP B：做一个新功能
1. 新开 session
2. 明确功能边界
3. 用 `@文件` 给最小充分上下文
4. 如果改动面大，先 planning
5. 如果逻辑复杂，先 thinking
6. 让 Claude 先提方案或计划
7. 人工审计划，再允许落地
8. 看 diff、跑测试、做手工检查

### SOP C：把高频任务固化成命令
命令模板建议包含：
- 任务目标
- 涉及目录
- 命名规范
- 测试要求
- 禁止扩散范围
- 必须引用的上下文文件
- 输出完成后的自检动作

### SOP D：引入 MCP
1. 先确定这是项目级还是个人级能力
2. 添加 MCP
3. 用 `/mcp` 检查状态
4. 在 prompt 里显式要求使用某个 MCP
5. 把成功经验写回项目 memory

### SOP E：引入 Subagents
1. 先定义职责单一的角色
2. 最小化工具权限
3. 让 agent 定义文件可被人工修改
4. 优先用于专项任务：测试 / 审查 / UX / 安全
5. 将其嵌入命令，而非靠手动临时想起

### SOP F：接入 GitHub
1. 先配好 `gh` 并登录
2. 安装 GitHub App / workflow
3. 小范围仓库授权
4. 用清晰 issue 试运行
5. 人工审 PR 和 review 结果

## 6. 最容易踩的坑

1. **一上来就把大任务全扔给 AI**
2. **把 `CLAUDE.md` 生成完就不维护**
3. **上下文越喂越多，导致噪声爆炸**
4. **Planning 过了就不再审边界**
5. **把自定义命令当免审通道**
6. **装了 MCP 就默认 Claude 会自动用**
7. **给 reviewer 型 subagent 太多权限**
8. **以为 GitHub 自动 review 等于万无一失**
9. **把 AI 当思维替身，而不是协作者**
10. **忽略版本差异和自动字幕误识别，把模糊信息当硬事实**

## 8. 哪些内容需要二次核验

由于本轮学习主要基于 metadata + 自动字幕提炼，以下类型信息不建议直接当硬规范照抄：

1. 个别模型名
2. 个别命令的连字符/空格/完整拼写
3. 个别快捷键字符
4. 个别 Windows 环境下的录制时 workaround

推荐做法：
- 原则吸收
- 细节落地时，以 Claude Code 当前版本 CLI/官方文档/实际补全为准

## 9. 最终压缩版结论

如果把整个系列压缩成最短的一段话，可以写成：

> Claude Code 的高质量使用方式，不是把一切都交给它，而是先把它接入你的终端和 Git 工作流，再用 `CLAUDE.md` 和精确上下文让它真正理解项目；随后通过权限控制、planning/thinking、slash commands、MCP、subagents 和 GitHub integration，把它逐步纳入一个可审查、可复用、可扩展、但始终由人类负责的开发系统。
