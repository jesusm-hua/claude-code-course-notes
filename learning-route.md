# Claude Code 系列教程学习路线

> 目标：不是“看完视频”，而是“形成可落地的 Claude Code 工作流能力”

---

## 0. 学习前提

开始前，你至少要有：
- 一个可试验的项目仓库
- 基础 Git 使用能力
- 能接受“AI 不是替你思考，而是帮你加速”这个前提

建议：
- 先在 throwaway project 或独立分支里练
- 从小任务开始，不要上来就做大功能

---

## 第一阶段：入门打底（建议 1 天）

### 学习内容
- #1 Introduction & Setup
- #2 CLAUDE.md Files & /init
- #3 Context

### 学习目标
你要真正掌握的是：
1. Claude Code 的定位是什么
2. `/init` 和 `CLAUDE.md` 为什么是底座
3. 上下文为什么会决定效果

### 学完后你应该能做到
- 在项目目录里启动 Claude Code
- 让它正确总结项目
- 执行 `/init`
- 手动补充一个像样的 `CLAUDE.md`
- 使用 `@文件`、选中片段、`/clear`、`/compact`

### 建议练习
1. 选一个小项目执行 `/init`
2. 给 `CLAUDE.md` 补 5 条项目规则
3. 做一个很小的任务，只给最小充分上下文
4. 完成后切换任务，练一次 `/clear` 或新开 session

### 验收标准
- 你能解释为什么“上下文不是越多越好”
- 你能区分 project 规则和个人偏好
- 你不会把 Claude Code 当全自动开发器

---

## 第二阶段：执行控制（建议 1 天）

### 学习内容
- #4 Tools & Permissions
- #5 Planning & Thinking

### 学习目标
你要真正掌握的是：
1. 哪些权限该放、哪些不该乱放
2. Planning 和 Thinking 的边界是什么
3. 为什么计划也需要 review

### 学完后你应该能做到
- 理解 read / edit / bash 的风险差异
- 为高频低风险命令做合理白名单
- 在“大范围修改”前先让 Claude 出 plan
- 在“复杂逻辑”前先让 Claude think

### 建议练习
1. 让 Claude 先 plan 一个多文件改动
2. 对 plan 提一次边界修正，再执行
3. 对一个复杂逻辑问题尝试 think / think hard
4. 对比“直接做”和“先计划后做”的差异

### 验收标准
- 你能说清 Planning 与 Thinking 的不同
- 你知道什么时候该继续审批、什么时候该允许连续编辑
- 你不会因为 Claude 给了 plan 就完全放弃判断

---

## 第三阶段：流程固化（建议 1 天）

### 学习内容
- #6 Slash Commands
- #7 MCP Servers

### 学习目标
你要真正掌握的是：
1. 怎么把高频 prompt 变成命令
2. 怎么让 Claude 查最新文档、做浏览器验收
3. 什么是 project / local / global scope

### 学完后你应该能做到
- 写一个项目级自定义命令
- 在命令中写清目标、目录、测试、边界
- 安装并验证一个 MCP
- 在 prompt 中显式要求使用某个 MCP

### 建议练习
1. 写一个“生成组件并补测试”的 slash command
2. 接入一个文档类 MCP（如 Context7）
3. 接入一个浏览器类 MCP（如 Playwright）
4. 写一条项目规则：实现第三方库功能前先查最新文档

### 验收标准
- 你能把重复工作变成可复用命令
- 你知道 MCP 不是“装完就完”，而是要验证、点名、写回规则
- 你能说清项目级配置和个人级配置的区别

---

## 第四阶段：角色分工与仓库协作（建议 1 天）

### 学习内容
- #8 Subagents
- #9 Claude Code with GitHub

### 学习目标
你要真正掌握的是：
1. Subagent 该怎么设计，才不会越拆越乱
2. GitHub 集成该如何在小范围里试跑
3. 为什么实现者与审查者最好分开

### 学完后你应该能做到
- 设计一个职责单一的 reviewer subagent
- 给它配置最小权限
- 把 subagent 嵌入某条命令或流程
- 在 GitHub 仓库中理解 issue -> branch -> PR -> review 的自动化链路

### 建议练习
1. 做一个 UX reviewer 或 test reviewer subagent
2. 让它只做审查，不给编辑权限
3. 用一个边界清晰的小 issue 试一次 GitHub 集成流程
4. 人工复核生成的 workflow 文件

### 验收标准
- 你不会把 subagent 当成“多开几个 AI 一起乱干”
- 你知道 reviewer agent 不该默认拥有写权限
- 你知道 GitHub auto review 不是最终判断

---

## 第五阶段：治理与长期习惯（建议半天）

### 学习内容
- #10 Final Thoughts & Tips

### 学习目标
你要真正形成的是长期习惯：
1. 新 feature 新 session
2. 始终看 diff、看代码、看产物
3. 不把 AI 当思维替身

### 建议练习
1. 故意把两个无关任务塞进同一 session，感受上下文污染
2. 再拆成两个干净 session，对比输出质量
3. 总结自己最容易失控的环节：权限、上下文、还是任务范围

### 验收标准
- 你形成自己的 session 管理习惯
- 你对“人始终在环”不再只是口号
- 你知道何时清理、何时重开、何时回退历史

---

## 推荐实战顺序（最落地版）

如果你想一边学一边真正搭出工作流，建议按下面顺序实践：

1. **先接入**：装好 Claude Code，进项目，跑 `/init`
2. **先记忆**：写好 `CLAUDE.md`
3. **先控上下文**：按 feature 管理 session
4. **再控权限**：理解哪些操作能放权
5. **再学计划**：大任务先 planning，难任务先 thinking
6. **再做命令**：把高频任务产品化
7. **再接 MCP**：先文档，再浏览器
8. **再做 subagent**：先 reviewer，再别的角色
9. **最后进 GitHub**：先小 issue，先小仓库授权

---

## 你学完后，理想状态应该是什么

不是“我会几个命令”，而是：

- 我知道 Claude Code 在我的工作流里该扮演什么角色
- 我知道如何给它稳定记忆和干净上下文
- 我知道如何限制它，而不是一味放权
- 我知道如何把一次性经验沉淀成命令和规则
- 我知道如何让它进入 subagent / GitHub / MCP 这些更高阶场景
- 我知道最终责任仍然在我，不在 AI

---

## 最后一句话

> 学 Claude Code，最重要的不是“看了多少集”，而是你有没有把它真正驯化成一个有边界、有流程、可复用、可审查的开发协作者。
