# 04｜Commands 和 MCP：怎么把一次性对话，变成可复用流程

很多人第一次用 Claude Code，会把它当成一个很强的聊天框。

这么用当然也行，但很快就会遇到两个问题：

1. 同样的话要重复说很多遍
2. 一旦涉及外部文档、浏览器、数据库，Claude 开始靠猜

这两件事，对应的就是这一章：
- **slash commands**：把高频 prompt 固化下来
- **MCP**：把 Claude 接到外部真实环境

---

## 先记住 3 句话

1. slash command 的价值不是少打字，而是**固化流程**
2. MCP 的价值不是多几个插件，而是**让 Claude 少猜**
3. 这两件事一起用，Claude 才会从“会聊天”变成“会按流程做事”

---

## 先说 slash commands：为什么值得写

如果你发现自己经常重复说这些话：
- 帮我生成组件
- 记得补测试
- 只改 preview page，不要扩散到别的页面
- 按项目命名规范来

那你其实已经在暴露一个信号：

> 这段 prompt 不该继续靠嘴说了，应该沉淀成命令。

视频里这一集最有价值的地方，就是把 slash command 从“快捷指令”提升成了“项目工作流资产”。

---

## 一个命令应该长什么样

命令文件本质上就是一段结构化 prompt。

我更建议一个命令至少写清：
- 目标是什么
- 可以改哪些地方
- 不要改哪些地方
- 命名怎么做
- 测试怎么补
- 完成后怎么验证

### 一个例子

比如你想做一个“生成 UI 组件”的命令，里面应该至少有这些信息：

```md
- Create a reusable component
- Add a matching test file
- Keep preview changes only in the preview page
- Do not modify unrelated routes
- Run tests before finishing
```

这类写法的重点不是优雅，而是明确。

Claude 最怕的不是约束多，而是边界含糊。

---

## 自定义命令最容易踩的坑

### 1. 只写“要做什么”，不写“不要做什么”
这是最常见的问题。

如果你只写“生成一个组件”，Claude 很可能会顺手：
- 加页面
- 加链接
- 改别的 UI

所以命令里一定要带边界。

### 2. 改完命令却忘了重启会话
视频里有个细节很实用：命令文件更新后，通常要重新加载会话，Claude 才会认到新内容。

### 3. 把命令当免审机制
命令只是把 prompt 标准化，不代表结果就不用看。

---

## 再说 MCP：它真正解决了什么问题

Claude Code 不接 MCP 的时候，很多事情只能靠它已有知识去猜。

比如：
- 某个框架最新文档怎么写
- 某个网站真实页面长什么样
- 某个数据库表结构现在是什么

MCP 的意义就在这里。

它不是简单给 Claude 多几个功能，而是让它通过标准化接口去接触真实外部环境。

---

## 这一集里最值得记住的两个 MCP

### Context7
最适合做：**查最新文档**。

这个场景特别重要。因为很多 AI 产出的问题，不是推理差，而是知识旧。

如果你在做：
- Tailwind
- Next.js
- 某个 SDK
- 某个第三方库

优先用文档型 MCP，通常比让模型凭旧记忆生成更稳。

### Playwright MCP
最适合做：**浏览器验收**。

这类能力很适合解决一个常见问题：

代码看起来对，不代表真实页面就对。

有了浏览器型 MCP，Claude 才有机会从“看代码”走到“看结果”。

---

## MCP 不是装完就完

这一点视频里讲得很实在。

接完 MCP 后，你至少还要做三件事：

### 1. 确认作用域
常见有：
- `project`
- `local`
- `global`

我的理解很简单：
- 项目专属能力，用 `project`
- 个人长期通用能力，用 `global`
- 临时试验，用 `local`

### 2. 看连接状态
别装完就开始怀疑 prompt。先确认它到底连上了没。

### 3. 使用时最好点名
比如：

```text
Use context7 to check the latest Tailwind docs.
```

或者：

```text
Use Playwright to inspect this page in the browser.
```

很多时候，你点名之后，Claude 的行为会稳定很多。

---

## 一个最小组合例子

这一章最值得上手试的，其实是把 commands 和 MCP 连起来。

比如你要做一个组件改造，可以这样设计：

### Step 1：用命令固化工作流
命令要求：
- 只改指定组件
- 保留 props 不变
- 补测试
- 不改别的页面

### Step 2：需要文档时调用 Context7
比如查组件库或 CSS 框架最新写法。

### Step 3：需要验收时调用 Playwright
让 Claude 去真实打开页面，而不是只停留在代码层。

这时候 Claude 才开始像一个流程化的开发助手，而不是随机发挥的聊天机器人。

---

## 最容易踩的坑

### 1. 命令太泛
命令越泛，Claude 发挥空间越大。

### 2. 以为装了 MCP 它就会自动用
很多时候还是得你明确说“用哪个 MCP”。

### 3. 连接失败直接怪模型
先看状态、作用域、配置，再看 prompt。

### 4. 有了 MCP 就放大自动化范围
能力变强，不代表边界就可以消失。尤其涉及外部副作用时，更要谨慎。

---

## 我的判断

第 6、7 集让我最认同的一点是：

> Claude Code 真正开始变强，不是在“回答更聪明”的那一刻，而是在你把它放进可复用流程、并让它接触真实外部环境的那一刻。

命令解决的是“重复说”的问题。

MCP 解决的是“凭空猜”的问题。

这两个结合起来，Claude Code 才会越来越像一个靠谱的工程协作者。
