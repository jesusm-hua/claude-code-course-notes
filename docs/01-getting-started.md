# 01｜先跑起来：Claude Code 到底该怎么开始

如果你是第一次用 Claude Code，先别急着看 MCP、subagents、GitHub integration。

第一步只有一件事：**先把 Claude Code 放进你的开发工作流。**

这一章只讲三件事：
- 它和常见 AI 编程工具有什么不同
- 第一次进项目该做什么
- 怎么避免一开始就用跑偏

---

## 先记住 3 句话

- Claude Code 更像**终端里的协作者**，不是编辑器里的聊天面板
- 它更适合做**边界清楚的小任务**，不是一上来接管整个项目
- 第一次进项目时，先让它**理解项目**，再让它**修改代码**

这三条如果没立住，后面很容易从“用 AI 提效”变成“用 AI 制造返工”。

---

## 它和 Cursor / Copilot 这类工具的差别

Claude Code 最明显的区别是：它主要工作在 **terminal** 里。

这件事带来的影响很直接：
- 它更自然地进入项目目录
- 更容易和 Git、测试命令、本地脚本配合
- 你不用为了 AI 改掉自己的 IDE

所以我更愿意把它理解成：

> 一个在终端里协作的 coding agent。

不是“问一句回一句”的助手，而是能真的读仓库、提命令、改文件、参与流程的工具。

---

## 第一次进入项目，我建议这样做

### Step 1：进入项目目录

```bash
cd your-project
```

### Step 2：启动 Claude Code

```bash
claude
```

第一次启动时，一般会让你完成：
- 登录
- 主题选择
- 是否信任当前项目目录

这里最重要的不是选项本身，而是 Claude Code 明确提醒你：
- 它会犯错
- 输出要 review
- 最好只在可信代码库里用

这不是套话，是真护栏。

### Step 3：先让它总结项目

先别说“帮我做功能”。

先问：

```text
Can you give me a quick summary of this project?
```

这一步很好用，因为你能马上判断：
- 它有没有读懂项目结构
- 它对技术栈理解对不对
- 它是不是在根据真实文件回答

如果这一轮总结都不靠谱，后面就更不该让它直接动代码。

### Step 4：开一个分支再试

```bash
git checkout -b chore/try-claude-code
```

原因很简单：Claude Code 会放大修改速度，也会放大犯错范围。

有分支，你就有回滚空间。

---

## 一个最小可执行练习

今天如果只做一个练习，我建议你先做“读项目”，不要急着做“改项目”。

### 输入

```text
Can you summarize the project structure, the main stack, and how to run it locally?
```

### 重点看什么

别只看它写得像不像样，重点看：
- 有没有提到真实目录
- 有没有说对运行命令
- 有没有把测试方式说错
- 有没有凭空脑补项目复杂度

### 可以继续追问

```text
Which files should I read first if I want to understand this project quickly?
```

这一步的价值很高，因为你是在先和它建立“共同地图”。

地图不对，后面越快越容易偏。

---

## 一个我不建议的起手方式

下面这种 prompt 很容易翻车：

```text
Build a complete auth system for this project.
```

不是因为 Claude Code 一定不会做，而是因为这类任务：
- 范围大
- 约束多
- 很容易牵连全局

如果项目语境还没立住，它就只能一路猜：
- 猜目录
- 猜路由
- 猜状态管理
- 猜数据层

最后生成一堆“看起来能用”的东西，但很可能不贴项目。

所以更稳的顺序是：

**先读，后改；先总结，后生成。**

---

## 我建议的起手顺序

1. 进入项目目录
2. 启动 `claude`
3. 完成基础登录和信任确认
4. 让它总结项目
5. 自己检查总结是否靠谱
6. 新建分支
7. 从小任务开始

刚开始适合的任务包括：
- 改一个按钮文案
- 调整一个组件样式
- 给一个 hook 补注释
- 帮你定位某个页面入口

范围越小，你越容易判断它到底是在帮你，还是在添乱。

---

## 我的判断

第一集最值钱的地方，不是安装命令，而是它帮你立住了一个正确姿势：

> Claude Code 不是拿来外包整个项目的，而是拿来放进终端和 Git 工作流里，做一个你能看见、能约束、能随时接管的协作者。

如果这一步走对了，后面学 `CLAUDE.md`、context、commands、MCP、subagents 才会顺。
