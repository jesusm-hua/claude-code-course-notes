# 01｜先跑起来：Claude Code 到底该怎么开始

如果你刚接触 Claude Code，我不建议一上来就看 MCP、subagents、GitHub integration。

那些东西当然重要，但都不是第一步。

第一步其实很朴素：**先把 Claude Code 安全地放进你的开发工作流里。**

这一章只解决三个问题：

1. Claude Code 跟 Cursor / Copilot 这类工具到底有什么不同
2. 第一次进入一个项目时，应该先做什么
3. 怎样避免一开始就把它用成“放养式 AI 编程”

---

## 先给结论

我看完这个系列的第一集之后，最想保留的判断只有三个：

- Claude Code 最像一个**终端里的协作者**，不是一个花哨的 AI 编辑器
- 它最适合做的是**边界清楚的小任务**，不是一股脑接管整个项目
- 第一次进项目时，最重要的不是“让它写东西”，而是**先让它理解项目，再让它动代码**

这个顺序很关键。

很多人第一次用这类工具，失败不是因为模型不够聪明，而是因为第一句就说：

> 帮我把这个功能做完。

Claude Code 当然会动，但它是在一个你还没跟它建立共同语境的状态下动。结果通常就是：

- 改了不该改的地方
- 做了你没要求的扩展
- 生成的代码看起来能跑，但跟项目风格不贴

所以别急着下大单。先把环境和语境搭起来。

---

## Claude Code 和常见 AI 编程工具有什么不一样

视频里的一个核心点，我觉得讲得很准：Claude Code 不是嵌在编辑器侧边栏里的那类工具，它主要住在 **terminal** 里。

这会带来两个直接差别。

第一，它更像开发流程的一部分，而不是编辑器里的一个聊天面板。你会更自然地把它放进：

- 项目目录
- Git 分支
- 测试命令
- 本地脚本
- review 流程

第二，它不会强迫你因为 AI 换掉原来的工作方式。你可以继续用自己的 IDE，只是把 Claude Code 当成一个在终端里协作的 agent。

这个区别看起来不大，实际影响很大。

如果一个工具只存在于聊天框里，你很容易把它当“问答助手”在用。

如果一个工具从项目根目录启动、会读代码库、会申请执行命令、会跟 Git 配合，它就更像一个真的在项目里工作的协作者。

## 第一次进入项目，我建议你这样做

别急着改文件。先跑一轮最小流程。

### Step 1：进入项目目录

```bash
cd your-project
```

### Step 2：启动 Claude Code

```bash
claude
```

第一次进来时，Claude Code 一般会让你完成一些基础设置，比如：

- 登录方式
- 主题模式
- 是否信任当前项目目录

这里最值得留意的，不是设置项本身，而是它给你的安全提醒：

- Claude 可能会犯错
- 输出要自己 review
- 最好只在你信任的代码库里使用

这不是免责声明，是真建议。

如果项目本身就不可信，或者你连里面的脚本/说明文件都没看过，直接让 agent 读和执行，很容易把风险放大。

### Step 3：先问它“这个项目在干什么”

这一句特别值得养成习惯。

你可以直接问：

```text
Can you give me a quick summary of this project?
```

我很喜欢这个动作，因为它能一次测出三件事：

1. 它有没有读懂项目结构
2. 它抓到的技术栈对不对
3. 它是在根据真实文件回答，还是在泛泛猜测

如果这一步的总结都明显不靠谱，后面就更不该急着让它直接改代码。

### Step 4：新建分支，再开始做事

这一点视频里虽然只是顺带提，但我认为应该上升成默认动作。

```bash
git checkout -b chore/try-claude-code
```

为什么？因为 Claude Code 不是不会犯错，它只是把“犯错的速度和范围”都放大了。

有分支，你就有回滚空间。

没有分支，第一次试就直接在主分支上乱改，后面很容易从“试工具”变成“收拾残局”。

## 一个最小实战：只做“读项目”，先别做“改项目”

如果你今天只想做一个最小练习，我建议不是写功能，而是这套：

### 你输入

```text
Can you summarize the project structure, the main stack, and how to run it locally?
```

### 你观察

重点别只看它写了多少，而是看下面这些：

- 有没有提到真实存在的目录
- 有没有说对运行命令
- 有没有把测试方式、样式方案、数据来源说错
- 有没有把一个普通项目脑补成“企业级系统”

### 你再追问

如果第一轮还行，可以继续问：

```text
Which files should I read first if I want to understand this project quickly?
```

这比直接让它“给我加个功能”更有价值。

因为这一步本质上是在建立共同地图。地图不对，导航越快，偏得越远。

## 我不建议的一种起手方式

下面这种方式看起来很省事，其实最容易翻车：

```text
Build a complete auth system for this project.
```

问题不在于 Claude Code 能不能写 auth，而在于你在项目语境还没建立时，就把一个跨度大、约束多、容易牵连全局的任务直接扔了出去。

通常会出这些问题：

- 它先猜项目结构
- 再猜路由方式
- 再猜状态管理
- 再猜数据库接入方式
- 最后再生成一堆“逻辑上说得过去”的代码

每一步都可能偏。

所以我的建议很简单：

**先让它读，后让它改；先让它总结，后让它生成。**

---

## 这一章最重要的实操建议

如果你已经准备今天就上手，我建议按这个顺序：

1. 进项目目录
2. 启动 `claude`
3. 完成基础登录和信任确认
4. 让它总结项目
5. 自己检查总结是否靠谱
6. 新建分支
7. 从一个很小的任务开始

比如这种任务就不错：

- 给某个按钮改文案
- 调整一个组件的 className
- 给某个 hook 补注释
- 帮你找出一个页面从哪里进入

范围越小，你越容易判断它到底是在帮你，还是在添乱。

---

## 我的判断

第一集最有价值的地方，不是安装命令本身，而是它立住了一个很重要的使用姿势：

> Claude Code 不是拿来“外包整个项目”的，而是拿来放进你的终端和 Git 工作流里，做一个你能看见、能约束、能随时接管的协作者。

如果第一步就把这个姿势立住，后面学 `CLAUDE.md`、context、commands、MCP、subagents 才会顺。

如果第一步就把它当成“更强的自动写代码工具”，那后面能力越多，通常只会越快失控。
