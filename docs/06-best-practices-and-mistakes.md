# 06｜最佳实践和常见误用：用久了之后，哪些习惯最重要

看到这一章，功能层面其实差不多已经串起来了。

最后真正决定你会不会长期用顺的，反而不是新功能，而是习惯。

这一章不讲新能力，主要收束两件事：
- 哪些做法值得长期保留
- 哪些误用最容易让 Claude Code 失控

---

## 我最认可的 8 条实践

### 1. 新项目先 `/init`
别省这一步。

它不是为了好看，而是为了给项目建立长期记忆底座。

### 2. `CLAUDE.md` 要维护，不是生成一次就结束
项目结构、规范、约束变了，`CLAUDE.md` 也该一起变。

### 3. 一个 feature 一个 session
这是我最推荐保留的习惯之一。

功能换了，会话最好也换。这样上下文最干净。

### 4. 大任务先 plan，难任务先 think
别把 planning 和 thinking 混用。

### 5. 命令能沉淀就沉淀
如果一段 prompt 会反复说，那它就不该只待在聊天里。

### 6. 接第三方能力时，优先查最新文档
MCP 最实用的用法之一，就是让 Claude 不再靠旧知识猜。

### 7. 实现和审查尽量分角色
这是 subagent 最值钱的地方。

### 8. 永远自己看 diff、看代码、看产物
这一条没有替代品。

Claude 可以帮你提效，但它不能替你负责。

---

## 我最不建议的 6 种用法

### 1. 一上来就让它做大功能
项目还没读懂，规则还没写清，直接做大任务，通常只会让返工更大。

### 2. 一个 session 里混很多主题
前半小时聊数据流，后半小时改 UI，Claude 很容易把旧目标带进来。

### 3. 看到 plan 就直接放心
plan 是方案草稿，不是正确性证明。

### 4. 把命令当免审机制
slash command 只是标准化 prompt，不代表结果天然更对。

### 5. 装上 MCP 就默认它会自动用
很多时候仍然需要你点名、检查状态、确认 scope。

### 6. 把 AI 当思维替身
这是长期最危险的一种用法。

短期效率可能很高，长期你会失去对代码库的真实控制感。

---

## 一个很简单的自检清单

每次准备让 Claude Code 干活前，我觉得都可以先过这 6 个问题：

1. 这个任务范围清楚吗？
2. 当前上下文干净吗？
3. 项目规则写清楚了吗？
4. 这次该 plan 还是该 think？
5. 这个动作的权限是不是放得太大了？
6. 做完之后谁来 review？

只要这 6 个问题里有 2 个答不上来，我通常就不建议直接开干。

---

## 如果只能保留一套最小工作流

如果你不想记太多，我建议至少保留这一套：

1. 进项目后先 `/init`
2. 写或补 `CLAUDE.md`
3. 一个 feature 一个 session
4. 改动大先 planning
5. 逻辑难先 thinking
6. 重要结果必须人工 review

只靠这 6 条，已经能避开很多常见翻车。

---

## 我的判断

Claude Code 真正难的地方，不是学会某个命令，而是建立一套不会越用越乱的工作习惯。

所以我现在更愿意把它理解成：

> 一个需要被管理的高能力协作者。

它不是不能做很多事，而是你必须一直知道：
- 它现在在做什么
- 为什么这么做
- 边界在哪
- 出问题时怎么收回来

这一点如果抓住了，Claude Code 会越来越顺手。

抓不住的话，就算功能再强，也很容易变成高成本试验品。
