# 05｜Subagents 和 GitHub：怎么把 Claude Code 从个人工具推进到协作系统

前面几章讲的，基本都还是“你和 Claude”之间的关系。

到了这一章，问题开始变成：

- 能不能让 Claude 分角色工作
- 能不能让它进入真实仓库协作流

也就是：
- **Subagents**
- **GitHub integration**

这两个能力一旦接对，Claude Code 就不只是本地工具了。

---

## 先记住 3 句话

1. subagent 不是“多开几个 AI”，而是**把职责拆开**
2. GitHub integration 不是“自动提交一切”，而是**把 AI 接入 issue / PR / review 流**
3. 越是进入正式协作流，越要保留人工复核

---

## 先说 subagents：到底为什么值得用

Claude Code 默认像一个主线程协作者：你说任务，它来做。

问题是，一旦任务变复杂，主线程很容易同时背太多东西：
- 实现逻辑
- 样式判断
- 测试补齐
- 安全审查
- 浏览器验收

这时候 subagent 的价值就出来了。

它不是为了热闹，而是为了把角色拆开。

最典型的拆法是：
- 一个负责实现
- 一个负责审查

这比“同一个 agent 一边写一边夸自己写得不错”可靠得多。

---

## 哪些任务适合 subagent

我更推荐把 subagent 用在“窄而深”的任务上，而不是大而泛的任务。

比如：
- 只做测试审查
- 只做安全审查
- 只做 UI / UX 评审
- 只做浏览器验收

这类任务有两个好处：
1. 角色边界清楚
2. 提示词更容易写准

视频里一个很好的例子，就是 reviewer agent 结合 Playwright 去看真实页面，而不是只看代码。

---

## 一个 reviewer subagent 应该怎么设计

我觉得有三个点最重要：

### 1. 职责单一
别又让它评审，又让它顺手改代码。

### 2. 最小权限
如果它的任务只是审查，那就别默认给编辑权限。

### 3. 能嵌进固定流程
最好的状态不是“想起来才调用”，而是默认流程里就有它。

比如：
- 生成组件
- 调 reviewer 看页面
- 根据反馈修正
- 再复审一次

这样 subagent 才不是摆设。

---

## 再说 GitHub integration：最有价值的地方在哪里

这一集最值得记住的，不是某个安装命令，而是它把 Claude Code 从“本地协作者”推进成了“仓库工作流的一部分”。

这意味着 Claude 可以开始进入：
- issue
- feature branch
- PR
- automated review

也就是说，AI 不只是帮你写一段代码，而是可以参与一条完整协作链路。

---

## GitHub 集成最适合什么场景

我不建议一开始就把复杂需求直接扔给 GitHub workflow。

它最适合的是：
- 边界清楚
- 验收标准明确
- 改动面不大

比如：
- 某个链接没加
- 某个按钮状态错了
- 某个小组件逻辑缺一块

这类 issue 很适合让 Claude 去自动处理。

但如果问题本身就很糊，比如“优化整个前端架构”，那放进 GitHub 自动流里通常不会更稳，只会更乱。

---

## 一个更稳的使用方式

我会更推荐这样理解 GitHub integration：

### 让 Claude 自动做这些
- 接小 issue
- 开分支
- 提 PR
- 跑自动 review

### 但人一定要保留这些判断
- 这个 issue 该不该交给 Claude
- PR 改动是否贴业务需求
- review 结果是否真的可信
- 最终要不要 merge

自动 review 很有价值，但它不是最终裁决。

---

## 这章最容易踩的坑

### 1. 以为 subagent 越多越好
不是。拆分是有成本的。

适合拆的是专项职责，不是把所有事都拆成多个 agent。

### 2. reviewer agent 权限太大
如果 reviewer 也能随便写文件，角色边界很容易乱掉。

### 3. 以为装完 GitHub App 就会自动接一切 issue
视频里已经说明了，很多时候还是需要明确触发。

### 4. 把自动 review 当最终审判
review 通过，不等于业务正确，也不等于没有隐藏风险。

### 5. 不看 workflow 文件
GitHub action 真正怎么触发、有什么权限、会做什么，都值得自己看一眼。

---

## 一个我更认可的成熟姿势

如果把前面几章连起来，我觉得比较成熟的用法会像这样：

1. 用 `CLAUDE.md` 立住项目规则
2. 用 clean context 控住当前任务
3. 用 planning / thinking 控住执行方式
4. 用 commands 固化高频流程
5. 用 MCP 让 Claude 接触真实外部环境
6. 用 subagents 拆分实现和审查
7. 用 GitHub 把这套流程接进团队协作

你会发现，subagents 和 GitHub 不是孤立能力，而是前面那套方法真正进入协作层之后的结果。

---

## 我的判断

第 8、9 集最重要的，不是功能数量，而是角色升级：

> Claude Code 开始从“你本地的一个工具”，变成“可以进入流程、参与协作、但依然需要人类持续监管的系统部件”。

一旦走到这一步，最重要的就不再是“它能不能做”，而是：
- 谁让它做
- 它在流程里做哪一段
- 做完之后谁来兜底

这才是成熟用法和尝鲜用法真正分开的地方。
