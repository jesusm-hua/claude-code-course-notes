## 一、总览：这三集到底在讲什么

这三集并不是在教“让 AI 全自动写完项目”，而是在建立一个更稳妥的 Claude Code 入门框架：

1. **先接入工具**：把 Claude Code 装进现有开发工作流，而不是围着一个新 IDE 重新建工作方式。
2. **再给稳定记忆**：用 `CLAUDE.md` 把项目结构、规范、偏好写清楚，让 Claude 少猜。
3. **最后管好上下文**：每次只给与当前任务强相关的 context，避免历史会话和无关文件污染判断。

这三步合起来，形成的不是“提示词技巧”，而是一个**可复用的 AI 协作开发工作流底座**。

# Episode 1 — Introduction & Setup

## Metadata

- 标题：Claude Code Tutorial #1 - Introduction & Setup
- 频道：Net Ninja
- 时长：10:10
- 上传日期：2025-08-21
- 链接：https://www.youtube.com/watch?v=SUysp3sJHbA
- 视频定位：课程开场、安装方式、登录方式、项目内启动方式、VS Code 协作方式、基础安全与 Git 分支准备

## 1) 一句话结论

**Claude Code 的核心价值，不是“又一个 AI 编辑器”，而是把 AI coding agent 放进终端与现有工程流里，让你在保持人工控制权的前提下协作开发。**

## 2) 核心知识点（5-10 条）

1. **Claude Code 是终端里的 agentic coding tool。**
   它和 Copilot、Cursor、Windsurf 这类嵌在编辑器里的工具不同，重点是直接驻留在 terminal 中工作。

2. **终端形态的优势是更容易接入既有开发流程。**
   讲者强调：你不需要因为 AI 工具而切换 IDE，它可以更自然地进入原来的 workflow。

3. **它不只做代码生成，还能分析、规划、编辑项目代码。**
   视频里把它描述为能 analyze / plan / write / edit code，而不是单一补全工具。

4. **它还能接入 GitHub workflow。**
   讲者提到可用于 PR code review、对 open issues 独立处理等自动化协作场景。

5. **讲者认为 Claude Code 对代码库的“理解感”较强。**
   这是他个人使用体验：生成代码更贴近具体项目，而不是千篇一律的通用答案。

6. **课程明确反对“纯放养式 vibe coding”。**
   讲者认为完全撒手让 AI 写，会带来 bug、粗糙代码和技术债，长期难维护。

7. **更推荐的做法是：人和 Claude 并肩开发。**
   把任务切得更窄、更具体；人工持续检查、随时手改，保持“人在回路中”。

8. **安装与登录只是起点，真正重要的是把它放进项目里运行。**
   视频中先在官网看安装命令与资费，再进入目标项目目录，运行 `claude` 开始项目级会话。

9. **首次使用时，Claude Code 会要求基础信任与权限确认。**
   包括主题模式、登录方式、对项目目录是否信任等；它也明确提醒“Claude 可能出错，要审查输出”。

10. **它具备 Git 操作能力，但默认会先展示命令并请求许可。**
    例如切分支、stage、commit、merge、resolve conflicts、查询 commit 差异等。

## 3) 方法论 / 工作流启发

1. **把 Claude Code 当“终端协作者”，不要当“外包程序员”。**
   它适合承担被定义清楚、范围收敛的开发动作，而不是接管整项目。

2. **先用它“读项目”，再让它“改项目”。**
   视频中第一次进入项目后，讲者先让它总结当前项目是什么，等于先建立共同语境。

3. **从一开始就保留人工可见性。**
   讲者喜欢在 VS Code 中一边看 Claude 改动、一边自己随时下潜到代码细节，这种“可视审阅 + 手动接管”的模式很重要。

4. **在正式改代码前先做环境护栏。**
   包括：启用更方便输入的终端设置、先切到新分支、最好在 throwaway project 或受版本控制的仓库里练习。

5. **AI 协作的第一原则是可回滚。**
   讲者反复强调：哪怕试验，也至少要用 version control，以便撤销不想要的改动。

## 4) 容易误用的点

1. **误以为 Claude Code 适合“整项目全自动开发”。**
   视频立场相反：更适合窄任务、边做边审查。

2. **忽视首次提示里的安全提醒。**
   视频明确提到：只在你信任的代码库上使用，并始终审查生成内容。

3. **不做分支隔离，直接在主分支试验 AI 改动。**
   这会放大试错成本，也破坏可回滚性。

4. **把 Git 权限能力理解成“可以无脑全放开”。**
   讲者展示的是：Claude 会提出 bash 命令并请求许可，而不是默认无限制执行。

5. **把 VS Code 扩展能力当成 Claude Code 的前提。**
   视频里更像是“增强体验”，不是“没有扩展就不能用”。

## 不确定 / 未明确说明

- 自动字幕把部分模型名识别成了类似 “Sonic 4 / Opus 4.1” 的写法；**具体模型拼写应以当时官方界面为准**。
- 安装命令的完整包名在自动字幕中识别不稳定；**可确定的是讲者从官网复制 npm 全局安装命令进行安装**。
- 讲者提到的 `/terminal-setup` 用于安装 `Shift+Enter` 新起一行的键位支持；**命令拼写建议以实际 CLI 补全/官方文档为准**。

# Episode 2 — CLAUDE.md Files & /init

## Metadata

- 标题：Claude Code Tutorial #2 - CLAUDE.md Files & /init
- 频道：Net Ninja
- 时长：12:17
- 上传日期：2025-08-22
- 链接：https://www.youtube.com/watch?v=i_OHQH4-M2Y
- 视频定位：解释 `CLAUDE.md` 的作用、`/init` 初始化流程、三层 memory 的区别，以及如何把项目规则写给 Claude

## 1) 一句话结论

**`CLAUDE.md` 的本质是“给 Claude 看的项目级运行说明书”，它把 AI 从“猜你的项目”变成“按你的项目规则做事”。**

## 2) 核心知识点（5-10 条）

1. **新接入 Claude Code 时，先运行 `/init` 是好习惯。**
   讲者把它视为进入一个项目后的标准初始化动作。

2. **`/init` 会扫描代码库并生成 `CLAUDE.md`。**
   它会查看目录结构、启动方式、状态管理、文档、`package.json` 等，然后输出结构化摘要。

3. **生成出的 `CLAUDE.md` 是人类可读的。**
   视频展示了其中可能包含：开发命令、架构概览、项目结构、数据架构、样式信息、测试设置、开发备注等。

4. **Claude 会自动把 `CLAUDE.md` 加入 session context。**
   这意味着之后你让它读代码、改代码，它会把这个文件当作常驻指导。

5. **老项目更容易生成“有内容的” `CLAUDE.md`。**
   因为已有代码、结构与配置可供扫描；新项目若几乎为空，自动生成信息会少很多。

6. **如果自动生成不够，就应手动补。**
   讲者建议手动写入高层项目结构、会使用的工具/框架、代码风格偏好、应用整体概述等。

7. **`CLAUDE.md` 不是一次性文件，而是要持续更新的。**
   如果文件结构、依赖、规范变了而你不更新它，Claude 可能仍按旧理解工作，越做越偏。

8. **你可以主动把新规则写进 `CLAUDE.md`。**
   视频中讲者手动补了一条：所有可复用 hooks 放进 `hooks` 文件夹，用来引导 Claude 的生成路径。

9. **可以用 `#` 快速记录 memory。**
   在聊天里输入类似“当创建新页面时，总在 header 加链接”，Claude 会把它作为 memory 写入某类 `CLAUDE.md` 文件。

10. **视频区分了三类 memory：project / local project / user(global)。**
    - project：项目共享规则，应纳入版本控制，可推到远端仓库。
    - local project：仅自己在这个项目里的偏好，不给团队共享。
    - user(global)：自己跨项目的通用偏好与工具习惯。

## 3) 方法论 / 工作流启发

1. **把 AI 上下文前置成“项目文档”，不要每次靠 prompt 临时解释。**
   `CLAUDE.md` 的价值在于把重复说明沉淀掉，让后续每次协作都站在同一基线之上。

2. **把结构性规范写死在 memory 里，比口头提醒更可靠。**
   比如 hooks 放哪、页面创建时是否要补导航、测试如何写、命名怎么定。

3. **优先写“项目约束”，其次才是“任务要求”。**
   任务 prompt 会变，但项目架构、目录、惯例、风格，应该通过 `CLAUDE.md` 稳定提供。

4. **共享规则和个人偏好要分层。**
   团队共识进入 project memory；个人习惯不要混进项目公共规范里。

5. **用一次真实小任务验证 memory 是否生效。**
   视频里先补 hooks 规则，再让 Claude 创建 hook；再写“新页面要加 header link”，随后创建 about page 验证它是否遵守。

## 4) 容易误用的点

1. **把 `/init` 生成的 `CLAUDE.md` 当最终版，不再维护。**
   这是视频里明确反对的做法。

2. **把“共享项目规则”和“个人偏好”写进同一个层级。**
   这会让团队协作边界混乱。

3. **相信 Claude 会自动从最新代码状态中完全推断规范变化。**
   讲者提醒：如果 `CLAUDE.md` 不更新，Claude 可能继续沿旧规则工作。

4. **不加约束地让 Claude 顺手完成额外动作。**
   视频里举例：它创建新 hook、页面时，常会顺手做更多事；要在 prompt 里明确“先别使用它”“只做这一部分”。

5. **把 memory 当成万能约束，而不做结果检查。**
   视频虽然演示了 memory 生效，但也展示了 Claude 有“超范围发挥”的倾向。

## 不确定 / 未明确说明

- 自动字幕把 `CLAUDE.md` 频繁识别成 `claw.md / clawed ND` 等变体；**可确认视频讲的就是 `CLAUDE.md` memory file**。
- 讲者提到“local memory 正在被 deprecated，推荐未来改成在 project memory 中 import 未跟踪文件”的做法；**这是视频转述 docs 的说法，未在本集展开具体导入语法**。
- `/memory` 命令被展示为可直接打开不同 memory 文件编辑；**具体命令展示明确，但不同版本 CLI 菜单项可能略有差异，建议以当下界面为准**。

# Episode 3 — Context

## Metadata

- 标题：Claude Code Tutorial #3 - Context
- 频道：Net Ninja
- 时长：13:03
- 上传日期：2025-08-25
- 链接：https://www.youtube.com/watch?v=ob-mYGqqFQw
- 视频定位：解释 context 是什么、如何加文件/代码/图片上下文、如何避免上下文膨胀，以及如何用命令管理 session context

## 1) 一句话结论

**Claude Code 的效果高度取决于你是否给了“刚刚好”的上下文：太少，它要瞎猜；太多，它会被历史噪声拖偏。**

## 2) 核心知识点（5-10 条）

1. **`CLAUDE.md` 之所以有用，是因为它会自动进入 session context。**
   这集先把“memory file”与“context”关联起来：前者之所以生效，是因为后者存在。

2. **上下文就是让模型更准确地理解你到底在说什么。**
   讲者用“banana”举例：信息不足时，词本身没意义；补上语境，回应就会更合理。

3. **可以用 `@路径` 手动把文件加入上下文。**
   在 prompt 中输入 `@` 后跟文件路径，Claude 会读取该文件作为任务背景。

4. **打开文件并把光标放进去，也能把当前文件加入上下文。**
   这是更轻量的“当前活动文件感知”方式。

5. **还可以只选中一段代码，把局部片段作为上下文。**
   这意味着你不一定总要给整文件。

6. **可以把图片拖进聊天，作为视觉上下文。**
   视频里就用按钮截图，让 Claude 按图片风格改按钮组件。

7. **也可以把配置/模式文件通过 `@` 引入，作为规则型上下文。**
   例如 schema、测试配置等，让 Claude 在编辑相关区域时更有依据。

8. **当前 session 的全部聊天历史、之前加入的文件、模型回答，都会累积成上下文。**
   这也是“越聊越乱”的根源。

9. **上下文膨胀会让结果变差。**
   如果你从购物车功能聊到按钮样式修改，旧的 basket 历史也可能继续影响当前按钮任务。

10. **视频给了多种 context 管理手段。**
    - `/clear`：清空会话历史与手动加入的 context，但保留 `CLAUDE.md` 这类基础 memory。 
    - `/compact`：把长历史压成简短摘要，保留摘要、删除其余细节。 
    - 双击 `Esc`：回退到之前的会话节点，裁掉后续历史。 
    - `/exit` 后重新 `claude`：开一个全新会话。 
    - 恢复旧会话的命令：字幕中像是 `/res`，功能是恢复之前 session。**命令名以实际 CLI 为准。**

## 3) 方法论 / 工作流启发

1. **默认应按“任务单元”管理会话，而不是一个会话聊完所有开发工作。**
   一个功能一个 session，或一个阶段一个 session，更容易保持清洁上下文。

2. **上下文要“最小充分”，不是“越多越好”。**
   只提供当前任务必需的文件、片段、图片、规则，通常比一股脑喂整个仓库更有效。

3. **先定位再提问，比笼统命令效果好。**
   比如“修这个按钮组件”配合 `@button.tsx`，远优于“帮我修一下样式”。

4. **当任务从功能实现切换到 UI 调整时，应考虑清理或重开会话。**
   这集最核心的实战启发，就是不要让旧 feature 的历史拖累新任务。

5. **`/compact` 更适合“同一任务继续做”，`/clear` 更适合“任务完成切下一个”。**
   这是视频呈现出来的最佳使用边界。

## 4) 容易误用的点

1. **把所有相关/不相关文件都塞进上下文。**
   这不一定更聪明，反而会增加噪声。

2. **长会话里不断换主题，却不清上下文。**
   这样 Claude 很容易把旧目标、旧文件、旧假设继续带进来。

3. **以为打开文件就等于精确定位问题。**
   如果问题只在少数几行，最好进一步选中代码片段，缩小上下文范围。

4. **把 `/compact` 当 `/clear` 用。**
   `/compact` 不是彻底清除，而是压缩保留；如果想从零开始，应使用 `/clear` 或新会话。

5. **忽视 context window 的上限。**
   视频提到录制时 Claude Code 的 context window 为 200,000 tokens；接近上限时，输出质量可能下降。

## 不确定 / 未明确说明

- 恢复历史会话的命令，自动字幕更像识别成 `/res`；**可确认有“恢复旧 session”的能力，但具体命令名请以 CLI 提示为准**。
- context window 被讲者说成 200,000 tokens，是“录制当时”的数值；**不同版本/模型未来可能变化**。
- 视频演示了拖图改样式，但没有系统讲图片理解的限制边界；**比如多图、复杂布局、多轮视觉迭代的最佳实践，本集未展开**。

# 四、这三集合起来形成的能力主线

## 1) 能力主线概括

这三集连起来，其实是在建立 Claude Code 的三层控制面：

1. **接入层**：让 Claude Code 进入你的终端、项目目录、编辑器协作环境与 Git 流程。  
2. **记忆层**：用 `CLAUDE.md` 把项目结构、规范、偏好、惯例稳定写给 Claude。  
3. **上下文层**：在具体任务里，精确控制 Claude 当前“看见什么、不看见什么”。

如果只做第 1 步，你只是“装上了工具”；
如果做到第 2 步，Claude 开始“懂项目”；
如果做到第 3 步，Claude 才真正进入“可控协作”的状态。

## 2) 三集串起来的进阶关系

- **第 1 集解决的是“能不能用起来”**：安装、登录、进项目、进 VS Code、进 Git 分支。
- **第 2 集解决的是“Claude 怎么少走弯路”**：把项目规则沉淀为 `CLAUDE.md`，减少反复解释。
- **第 3 集解决的是“为什么同样的 Claude 有时表现好、有时跑偏”**：关键在上下文质量，而不是只看 prompt 文案。

## 3) 最核心的底层认识

1. **Claude Code 不是魔法，它的表现取决于你给它的结构化信息质量。**
2. **真正能放大产出的，不是一次神 prompt，而是稳定的项目记忆 + 干净的任务上下文。**
3. **AI coding 的关键不是“放权”，而是“设边界 + 持续审阅 + 可回滚”。**
4. **Claude 更像一个高能力、但需要管理的协作者；不是一个不需要管理的自动工人。**

# 五、给主控的 handoff：最值得纳入最终知识文档的点

以下是我认为最值得进入最终知识文档、且可复用性最高的内容：

## A. 应该沉淀成“固定工作流”的点

1. **新项目接入 Claude Code 的标准起手式**
   - 进入项目目录再启动 `claude`
   - 先让它总结项目，再开始改代码
   - 先切独立分支，再进行 AI 改动
   - 始终保留人工 review 与回滚能力

2. **`/init` 应进入项目 SOP**
   - 新项目或新接入 Claude Code 时，默认执行 `/init`
   - 生成 `CLAUDE.md` 后立即人工审阅并补齐缺失规范
   - 后续项目结构变化时同步更新 `CLAUDE.md`

3. **`CLAUDE.md` 的内容分层要明确**
   建议最终知识文档明确写出：
   - 项目结构
   - 开发/测试/运行命令
   - 关键架构约束
   - 命名规范
   - 组件/Hook/数据/样式/测试约定
   - “默认不要做什么”的边界条件

4. **把“共享规则”与“个人偏好”分离**
   project memory 与 user/global memory 不应混写；这对多人协作尤其重要。

## B. 应该沉淀成“提示策略”的点

5. **任务 prompt 要尽量窄、可验证、可停止**
   例如明确写：
   - 只创建，不接入
   - 只改某个文件/组件
   - 不要顺手做额外重构
   - 完成后先展示 diff / 方案

6. **把高频要求上移到 memory，不要每次重复说**
   例如：
   - 新页面是否要补导航
   - 新 hook 放哪里
   - 默认测试要求
   - 组件命名与目录规范

## C. 应该沉淀成“上下文管理准则”的点

7. **上下文管理是 Claude Code 实战效果的关键变量**
   最值得写进最终知识文档的一句话是：
   **“上下文不是越多越好，而是越贴近当前任务越好。”**

8. **推荐明确记录 5 种上下文管理动作及适用场景**
   - `@文件`：精确指定任务输入
   - 打开当前文件 / 选中片段：局部聚焦
   - `/compact`：同任务长会话压缩
   - `/clear`：任务切换时清场
   - 新开 session / 恢复旧 session：按 feature 管理上下文

9. **建议把“一个 feature 一个 session”写成默认实践**
   这能显著降低上下文污染，尤其适合复杂仓库。

## D. 应该沉淀成“风险提醒”的点

10. **不要把 Claude Code 当无人监管的自动开发器**
    视频的一条主线很明确：AI 可以提效，但不应跳过人工判断。

11. **Claude 有超范围发挥倾向，提示里要写边界**
    如果不写“不要顺手改别的”，它可能会做额外修改。

12. **自动化能力越强，越需要版本控制与审查纪律**
    Git 分支、diff 审查、权限确认，是这类工具的基本护栏。

# 六、可直接复用的结论版摘要

如果要把这三集压缩成一段最短可传播知识，可以写成：

> 用 Claude Code 做项目开发，先别急着让它写功能。先把它接入你的终端与 Git 工作流，再用 `/init` 生成并维护好 `CLAUDE.md`，把项目结构和约束明确写给它；接着在每次任务里只给最小充分的上下文，并在任务切换时及时清理 session。真正决定效果的，不只是提示词，而是**项目记忆是否清晰、当前上下文是否干净、人工审查是否持续在线**。
