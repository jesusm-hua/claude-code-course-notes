## 第 4 集：Tools & Permissions

### 1) 一句话结论
Claude Code 的核心不是“你手动选工具”，而是“它会自动挑工具”，你真正要学会的是：**哪些操作需要授权、哪些授权可以会话级或项目级放行，以及何时不该无脑放权**。

### 2) 核心知识点
1. Claude Code 会根据你的自然语言指令，自动调用内置工具，不需要你手写工具名。
2. 视频举的典型内置工具包括：`read`、`edit`、`bash`，以及用于自我跟踪任务的小清单工具（to-do）。
3. 不同工具的权限要求不同：视频明确提到 `read` 不需要授权，而 `edit`、`bash` 需要授权。
4. 当 Claude 想修改文件时，会先弹出审批；你可以批准、拒绝，或在某些场景下选择“不再重复询问”。
5. 简单的多处编辑也可能触发多次确认，因为它可能分多步修改同一个文件。
6. 对 bash / 命令类操作，可以把某类命令加入允许列表，后续在该项目里重复执行时不必每次确认。
7. 视频提到允许列表会写入项目本地的设置文件 `settings.local.json`；**具体目录名自动字幕识别不稳定，未在本笔记中强行写死**。
8. 这些项目本地允许规则更像“个人工作流偏好”，视频明确说这类文件**不应被提交并推送**。
9. 还可以开启“accept edits”模式，让当前聊天会话里的文件编辑不再逐次确认；新会话会重新恢复正常确认流程。
10. 视频口述快捷键：Windows 是 `Alt + M`；Mac 端他说“**可能**是 `Control + M`”，这里属于**未完全确认**的信息。

### 3) 方法论 / 工作流启发
- **先理解权限边界，再追求速度。** Claude 能做很多事，但默认审批是防呆机制，不是阻碍。
- **把高频、低风险命令做项目级白名单。** 例如测试、常规 git 命令，可以减少重复确认。
- **把“会话级自动接收编辑”和“项目级命令白名单”分开理解。** 前者偏临时提速，后者偏工作流配置。
- **读操作默认放开，写/执行操作重点审。** 这是最符合安全直觉的使用方式。

### 4) 容易误用的点
- 把“Claude 会自动选工具”理解成“所以我不用看它要干什么了”——错，越是写入/执行越要审。
- 一上来就给过宽权限，尤其 bash 命令白名单过大，后面容易失控。
- 把本地权限设置当成团队共享配置提交上去；视频明确不建议这么做。
- 开了 accept edits 后忘了自己已经处在“低摩擦写入”模式，导致审查松懈。

## 第 5 集：Planning & Thinking

### 1) 一句话结论
**Planning 解决“任务跨度大、步骤多”的问题，Thinking 解决“逻辑复杂、需要先想清楚”的问题；两者不是一回事。**

### 2) 核心知识点
1. 视频开宗明义区分两件事：
   - **Thinking**：先思考方案，再写代码。
   - **Planning**：先生成执行计划，再让你审批是否按这个计划落地。
2. Planning 模式适合“改动面广、涉及多个文件”的任务；单点逻辑未必很难，但改动范围比较大。
3. 视频演示里，Claude 会先扫描代码库、定位适用位置，再给出完整计划，包括新组件放哪、替换哪些现有实现、是否补测试、改哪些页面。
4. 对计划的处理不是只有“接受/拒绝”二选一；你可以接受计划并自动接收编辑，也可以继续人工审批，或者让它按反馈重做计划。
5. 计划模式的价值不只是“列清单”，更在于让 Claude 在多步骤任务里更不容易跑偏。
6. 但即使有计划，AI 也可能做“超出你原意”的事情。视频案例里，它额外创建了 avatar 页面并加到 header，作者认为这有点“过于积极”。
7. 作者随后修正项目里的规则文件（自动字幕把文件名识别成了类似 `claw.md` 的形式，**未完全确认**），补充更明确约束，说明：给新页面加 header 链接只适用于 page component，不适用于普通 UI 组件。
8. 这说明：**计划模式不能替代精确约束**；说明不清，AI 仍会按“它理解的合理方案”扩张执行。
9. Thinking 模式更适合“范围未必大，但逻辑复杂”的任务，例如带认证、数据库、审核、实时更新的评论系统。
10. 视频中通过在提示词里加入 `think` / `think hard` 来触发更强思考；还提到有不同强度：`think`、`think hard`、`think harder`、`ultra think`。
11. 思考强度越高，token 消耗越大，耗时也更长。
12. 作者明确提醒：不要把复杂大任务一次性交给 AI 放飞，最好拆成小块，保持自己一直在 loop 里。

### 3) 方法论 / 工作流启发
- **宽任务先 planning，难逻辑先 thinking。** 这是本集最值得固化成团队规则的一点。
- **计划先审，再执行。** 计划文本本身就是第一次 review。
- **AI 跑偏时，优先修 prompt / 项目规则，而不是只做一次性口头纠错。** 作者直接补充项目记忆/规范文件，就是在降低同类偏差复发。
- **复杂系统按 checkpoint 拆解。** 比如数据库、认证、实时同步、审核分别单做，而不是一次性全丢给 AI。
- **不要掉进“全自动 vibe coding”模式。** 作者态度很明确：人必须保持在闭环里。

### 4) 容易误用的点
- 把 Planning 当成更强版 Thinking，或反过来；两者解决的是不同问题。
- 看完计划就秒点通过，没有核对边界条件，最后 AI 在“合理扩展”里越做越多。
- 让 AI 一次性实现完整系统，再靠补锅式 prompt 修复，后续维护成本反而更高。
- 只记得“think 能更聪明”，忘了 token 和时间成本。

## 第 6 集：Slash Commands

### 1) 一句话结论
Slash command 的本质是：**把一段可复用的高质量 prompt + 上下文 + 约束，封装成项目级快捷入口**。

### 2) 核心知识点
1. Claude Code 自带不少 slash commands；视频举例提到 `/clear`、初始化项目记忆/规则文件的命令，以及模型切换、权限管理、子 agent 管理等命令。
2. 输入 `/` 可以看到可用命令列表，并可上下选择。
3. 作者重点不是讲内置命令，而是讲**自定义命令**：把项目里的高频任务固化成命令。
4. 自定义命令的放置方式是：在项目的 Claude 配置目录里建一个 `commands` 子目录，并放一个 Markdown 文件；文件名对应命令名。**自动字幕对目录名和后缀识别都不太稳，所以这里保留“Claude 配置目录 + commands 子目录 + Markdown 文件”这层事实。**
5. 命令文件本质上就是一段大 prompt：你可以写目标、目录约束、命名规范、测试要求、不要做什么等。
6. 在命令文件里也可以用 `@文件路径` 的方式引入上下文，例如 CSS 文件、规范文件等。
7. 作者示范了一个 UI 组件生成命令：要求生成组件、测试文件、运行测试直到通过，并把预览只加到 preview page，而不是乱加到别处。
8. 新建或修改完命令文件后，需要**重启 Claude 会话**，它才会重新加载这些命令。
9. 如果命令不带参数，Claude 会替你补全“它觉得合适”的内容；视频第一次运行时，它自己决定生成 `Card` 组件。
10. 因此更稳的做法是加参数：作者用 Markdown front matter 里的 `description` 和 `argument-hint` 给命令补元信息。
11. 运行时传入的参数会被 Claude 收进一个 `$ARGUMENTS` 值里；作者演示了把它进一步拆成 name / summary 两部分使用。
12. 作者特别说明：自己**没看到官方对多参数拆分的明确指导**，而是用一种“可用但偏技巧性”的办法处理，所以这部分应视为**经验法**，不是官方标准。
13. 演示调用方式里，他用类似“名称 + 分隔符 + 描述”的方式传两个值，之后 Claude 就能按指定名称和摘要生成组件。
14. 生成结果依然要人工 review；视频里作者就对“用 emoji 当图标”不满意，随后又要求 Claude 改用 icon library。

### 3) 方法论 / 工作流启发
- **把重复出现的好 prompt 产品化。** 不要每次重新口述同一套规则。
- **命令里要同时写“要做什么”和“不要做什么”。** 例如“只加到 preview page，不要扩散到项目其他地方”。
- **参数化 > 纯模板化。** 命令如果能吃参数，复用价值会高很多。
- **命令是团队规范载体。** 命令文件里沉淀的，其实是可执行的开发约束。
- **自定义命令不是免审通道。** 它只是把 prompt 标准化，不是把判断外包给 AI。

### 4) 容易误用的点
- 把自定义命令当成“万能脚本”，结果 prompt 太泛，Claude 自己发挥过多。
- 修改命令后忘记重启会话，以为命令没生效是 Claude 的问题。
- 过度依赖参数解析技巧；视频自己都承认多参数拆分方法并非官方明确推荐。
- 命令生成的内容不复查，尤其 UI/交互细节很容易“能跑但不够好”。

## 第 7 集：MCP Servers

### 1) 一句话结论
MCP 的真正价值不是“给 Claude 多几个插件”，而是：**让 Claude 通过标准化工具接口，安全地接入外部文档、浏览器、数据库、API 等真实世界能力。**

### 2) 核心知识点
1. MCP 是 `Model Context Protocol`，视频说它由 Anthropic 设计，用来定义 AI 模型如何与外部资源交互。
2. Claude Code 自带工具主要局限在当前代码库内；没有 MCP 时，它无法自然地操作外部数据库、第三方服务或最新在线文档。
3. MCP server 的本质是“提供一组外部能力工具”的中间层；**不是 Claude 直接连外部，而是服务器去连，Claude 调用服务器暴露出来的工具。**
4. 视频举了 Supabase MCP 的例子：可以列出表、执行 SQL、部署 edge functions 等；Claude 因此可以基于真实数据库结构生成接口或代码。
5. 还举了两个重点服务器：
   - **Context7**：拿最新框架/库文档；
   - **Playwright MCP**：开浏览器、点页面、看元素、截图。
6. 作者提到可以去 `pulsemcp.com` 之类的网站找更多 MCP server。
7. 安装方式是 `claude mcp add ...`；视频区分了**本地连接**和**远程连接**两类方式。
8. 作用域有三种：`project`、`local`、`global`。视频说默认是 `local`，但作者个人更常用 `project` 或 `global`。
9. `project` 作用域下，配置会写到项目根目录的 `mcp.json` 中；团队成员也更容易共享这套项目级 MCP 配置。
10. 对 Context7，作者先尝试本地连接；在 Windows 录制环境下遇到连接问题，于是改用了远程 HTTP 方式。
11. 视频明确提醒：Windows 下加 MCP 可能比 Mac 更“折腾”，尤其是 `npx` 场景，要额外加 `cmd /c`；而且某次还因为 `-y` 参数报错，最后靠手改配置补回去。
12. 这些 Windows 细节都是**录制当时的经验现象**，作者自己也说希望未来会更顺。
13. 安装好后，可以用 `/mcp` 查看连接状态；绿色勾表示已连通。视频里 Context7 甚至重连了一次才成功。
14. 真正使用时，最好在 prompt 里**显式点名**要用哪个 MCP，例如“Use context7”，这样 Claude 更明确会调用对应工具。
15. 作者实际用 Context7 检查 Tailwind 最新文档，确认项目里的 theme variables 配置是否正确。
16. 作者还把“实现第三方框架/库功能时优先用 Context7 查最新文档”写入项目 memory，避免 Claude 落回旧知识。
17. Playwright MCP 的演示是让 Claude 打开 `netninja.dev` 并总结网站内容，证明浏览器型外部能力已经接通。
18. 作者进一步点出更重要的用途：以后可让 Playwright 帮忙看组件在浏览器里的实际 UX，而不只是做静态代码生成。

### 3) 方法论 / 工作流启发
- **MCP 让 Claude 从“只会改本地代码”升级成“能接外部真实环境的开发助手”。**
- **优先把“最新文档访问”做成默认能力。** 这能显著降低 AI 产出过时代码的概率。
- **把 MCP 使用习惯写进 memory / 项目规则。** 不要每次靠临时提醒“去查最新 docs”。
- **根据复用范围选 scope：**
  - 项目专属能力 → `project`
  - 多项目通用能力 → `global`
  - 只给自己临时用 → `local`
- **把 `/mcp` 状态检查纳入排障动作。** 先看连没连上，再判断提示词问题。
- **浏览器型 MCP 不只是“能打开网页”，更适合做真实体验验证。**

### 4) 容易误用的点
- 装上 MCP 就默认 Claude 会自动用；实际上很多时候你最好明确说“Use context7 / use Playwright”。
- 连接失败时直接怪模型，而不是先看 `/mcp` 状态、scope、配置文件和 transport 类型。
- 把录制时的 Windows workaround 当成永久标准答案；作者自己都强调这块变化很快。
- 有了 MCP 就无限放大自动化范围，忽略权限、外部副作用和连通性稳定性。

## 这四集合起来形成的能力主线

这 4 集拼起来，其实是在搭一条非常清晰的 Claude Code 能力升级路径：

1. **第 4 集：先管住手**  
   学的是工具边界与权限控制——Claude 能做什么，什么时候需要你批准。

2. **第 5 集：再学会先想再做**  
   学的是在执行前引入计划和思考，让任务拆解更清楚，降低“直接乱改”的概率。

3. **第 6 集：把高频套路固化**  
   学的是把成熟 prompt / 约束 / 上下文打包成可复用命令，让工作流从“聊天”升级为“可重复执行的流程”。

4. **第 7 集：把能力接到外部世界**  
   学的是通过 MCP 把 Claude 接上最新文档、浏览器、数据库、服务 API，让它不再只困在本地代码库里。

### 汇总成一句主线
**Claude Code 的正确进阶顺序不是“先让它更自动”，而是“先控权限 → 再控规划 → 再固化流程 → 最后扩展外部能力”。**

这条顺序很重要，因为如果一开始就给它很多外部能力、很多写权限，但没有计划、命令模板和项目规则，最终大概率会滑向“更快地失控”。

## 给主控的 handoff：最值得纳入最终知识文档的点

以下是我认为最值得沉淀成团队级知识/规则的内容：

### A. 应该直接写成原则的点
1. **Planning 和 Thinking 必须区分使用场景**  
   - 宽范围、多文件任务 → Planning  
   - 窄范围、复杂逻辑任务 → Thinking
2. **默认保持人在 loop 里**  
   不要把大系统一次性交给 AI 放飞，优先拆 checkpoint。
3. **权限策略分层**  
   - 读：低风险  
   - 编辑：中风险，建议保留审查  
   - bash / 外部能力：高风险，按命令白名单或 scope 控制
4. **把高频 prompt 产品化为 slash commands**  
   自定义命令应视为“项目工作流资产”，不是一次性技巧。
5. **实现第三方库/框架功能时，默认要求查最新文档（如 Context7）**  
   防止模型回落到过时代码模式。

### B. 适合写成操作手册 / SOP 的点
1. **自定义命令的推荐结构**  
   目标 + 文件位置 + 命名规范 + 测试要求 + 禁止扩散范围 + 需要引用的上下文文件。
2. **新增/修改命令后必须重启 Claude 会话**。
3. **MCP 接入后先做 `/mcp` 健康检查**，再正式使用。
4. **使用 MCP 时在提示词中显式指定工具名**，例如 “Use context7”。
5. **项目级 MCP 配置优先使用 `project` scope**，这样更利于团队协作和配置可见性。

### C. 适合写成风险提示的点
1. **AI 会“合理扩张”任务边界**，哪怕已经有 plan。计划审批不能走过场。
2. **自定义命令不是免审机制**；它只是标准化 prompt，输出依然要 review。
3. **Windows 下 MCP 配置细节变化快**；不要把某个 workaround 当长期真理。
4. **自动字幕不可靠的路径/参数细节，最终应以官方 docs 为准。**

### D. 我建议主控最终知识文档重点突出的一句话
> **Claude Code 的高质量使用方式，不是把一切都交给它，而是用权限、计划、命令模板和 MCP 把它约束在一个可审、可复用、可扩展的工作流里。**
